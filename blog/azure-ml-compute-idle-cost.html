$11,400/Month in Idle Azure ML Compute:
How One Setting Keeps the Meter Running

An Azure Machine Learning compute cluster with min_nodes: 2 and no training jobs for 23 days. The team believed the cluster was “off.” It wasn't. CleanCloud found $11,400 in monthly waste on the first scan.

The Cluster Nobody Thought Was Running

A healthcare AI team finished a model training run in early March. The training completed successfully, the model was promoted to production, and the team moved on to the next sprint. Three weeks later, a platform engineer was reviewing Azure spending ahead of the quarter-end budget review. Something looked wrong in the ML workspace line items.

She ran CleanCloud against their Azure subscriptions. The first finding hit immediately:

The cluster was named gpu-training-cluster. It had been provisioned eighteen months earlier for a large-scale computer vision model. At the time, the team set min_nodes: 2 to reduce job startup latency — keeping two nodes warm between training runs. That made sense when training was happening weekly. After the model shipped to production, training frequency dropped to monthly. But the cluster stayed at min_nodes: 2.

For twenty-three days, the cluster sat idle. Eight V100 GPUs — two nodes, four GPUs each — running 24 hours a day, billing continuously. The team thought the cluster was “paused.” Azure ML compute doesn't have a pause. It has min_nodes: 0.

Why Azure ML Compute Billing Is Counterintuitive

Azure Machine Learning compute clusters are designed for elasticity. They can scale from zero nodes to hundreds and back to zero automatically as jobs come and go. The key parameter is min_nodes.

When min_nodes is set to 0, the cluster scales to zero when no jobs are running — no compute, no billing. When min_nodes is set to any number greater than zero, that many nodes stay provisioned permanently, regardless of job activity. Azure bills for those nodes continuously at the full VM rate.

min_nodes settingBehaviour when idleBilling when idle
min_nodes: 0 Scales to zero — all nodes deallocated ✓ Zero billing when idle
min_nodes: 1 1 node always running ✕ Full VM cost for 1 node, 24/7
min_nodes: 2 2 nodes always running ✕ Full VM cost for 2 nodes, 24/7

This is not a bug. It's the intended behaviour — warm nodes reduce job startup latency from 3–5 minutes to seconds. But it's a billing behaviour that most data scientists and ML engineers don't fully internalise when they provision the cluster. The Azure Portal shows the cluster as “Running” whether it's active or idle. There's no visual distinction.

The key insight: An AML cluster with min_nodes > 0 is always billing, even when idle. The only signal that it's wasted spend is the absence of recent job activity — which is what CleanCloud checks.

How the Detection Rule Works

The azure.ml.compute.idle rule checks three conditions for each compute cluster in each Azure ML workspace:

  1. min_nodes is greater than zero (billing continuously regardless of activity)
  2. No active, queued, or recently completed training jobs in the past N days (default: 7)
  3. The cluster is in Succeeded or Running state (not already stopped)

All three conditions must be met. A cluster with min_nodes: 0 that's idle isn't flagged — it's already billing zero. A cluster with min_nodes: 2 running an active job isn't flagged — the billing is justified. Only the combination of “can't scale to zero” + “no recent activity” = flagged.

Confidence is rated HIGH for GPU-backed clusters (the cost impact is unambiguous) and MEDIUM for CPU-only clusters where the cost may be modest.

The Fix: One Line of Config

Once the finding was confirmed, the fix took thirty seconds. The platform engineer updated the cluster configuration to set min_nodes: 0. The two warm nodes were deallocated by Azure within minutes. Billing dropped to zero for the cluster.

Azure CLI — fix idle cluster billing scale to zero when not in use 
# Scale minimum nodes to zero — cluster deallocates when idle
az ml compute update \
  --name gpu-training-cluster \
  --workspace-name ml-prod \
  --resource-group ml-platform-prod \
  --min-instances 0

The cluster kept its max_nodes: 10 configuration unchanged. The next training job still provisions nodes as needed — it just takes 3–4 minutes longer to start than when nodes were pre-warmed. The team decided that tradeoff was worth $11,400/month.

If you need low-latency startup but don't want to pay for permanent warm nodes, consider Azure ML's serverless compute for training, or set min_nodes: 1 only during active development sprints and min_nodes: 0 in production when training is infrequent.

Required Azure Permissions

The rule needs read-only access to Azure ML workspaces and their compute resources. The CleanCloud service principal needs the AzureML Data Scientist role (read-only variant) or an explicit assignment of:

  • Microsoft.MachineLearningServices/workspaces/read
  • Microsoft.MachineLearningServices/workspaces/computes/read
  • Microsoft.MachineLearningServices/workspaces/jobs/read

Frequently Asked Questions

Does Azure Machine Learning charge for idle compute clusters?

Yes. Clusters with min_nodes > 0 keep that many nodes provisioned continuously, billing at the full VM rate regardless of job activity. Setting min_nodes: 0 allows the cluster to scale to zero and stop billing when idle.

How do I stop an AML cluster from billing when idle?

Set min_nodes: 0 via the Azure Portal, Azure CLI (az ml compute update --min-instances 0), or the Python SDK. The cluster will deallocate all nodes when no jobs are running and provision them again when a new job is submitted.

Will setting min_nodes to 0 affect my training jobs?

Yes — job startup time increases from seconds (with warm nodes) to 3–5 minutes (cold cluster). For most teams running infrequent training jobs, this tradeoff is easily worth the cost saving. For latency-sensitive workflows, consider scheduling jobs to run in batches and setting a short idle timeout rather than permanent warm nodes.
