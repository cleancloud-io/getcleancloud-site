The Hidden Cost of
Idle AI/ML Infrastructure

SageMaker endpoints don't stop billing when traffic stops. Azure ML compute clusters keep running when training finishes. Vertex AI endpoints sit idle for weeks after a proof-of-concept wraps up. The bill arrives, and nobody knows why it's so high.

AI/ML Waste: The New Blind Spot

For the past several years, the dominant form of cloud waste was infrastructure sprawl — orphaned EBS volumes, idle NAT gateways, stopped VMs that were never deallocated. These resources are expensive in aggregate, but individually small. A forgotten EBS volume costs $10–$40/month. An unattached Elastic IP costs pennies.

AI/ML infrastructure operates at a completely different scale. A single idle SageMaker endpoint backed by a GPU instance can cost more than a thousand stopped EC2 instances. A Vertex AI endpoint running an undeployed model on a high-memory accelerator can burn through $23,000 a month with zero traffic. This is the new waste category that FinOps dashboards were never designed to catch.

The pattern is consistent across all three major clouds. A data scientist spins up an endpoint to serve a model — for a hackathon, a proof-of-concept, a demo. The event ends. The endpoint doesn't. Six weeks later, finance flags an anomaly in the ML budget. By then, the team has scattered, the model is stale, and the endpoint has cost more than the original experiment was worth.

“The proof-of-concept finished in March. The SageMaker endpoint was still running in August. Nobody remembered who created it.”

CleanCloud's AI/ML waste rules cover all three major clouds: SageMaker (AWS), Azure Machine Learning compute clusters (Azure), and Vertex AI prediction endpoints (GCP). All three use the same detection approach — measure real invocation traffic over a configurable idle window and flag instances with zero or near-zero activity.

AWS: Idle SageMaker Endpoints

SageMaker endpoints are always-on serving infrastructure. Unlike Lambda or Fargate, an endpoint doesn't scale to zero when traffic stops. The underlying EC2 instance keeps running, and AWS keeps billing — at full on-demand rates, including any attached GPU capacity.

How the rule works

The aws.sagemaker.endpoint.idle rule queries CloudWatch Metrics for InvocationCount across the idle window (default: 7 days). An endpoint with zero invocations over that period is flagged. The confidence level and cost estimate depend on the instance type:

Instance type Approx. idle cost/month Confidence
ml.t3.medium ~$30 MEDIUM
ml.m5.xlarge ~$140 MEDIUM
ml.g4dn.xlarge ~$500 HIGH
ml.g4dn.12xlarge ~$1,800 HIGH
ml.p3.2xlarge ~$2,800 HIGH
ml.p4d.24xlarge ~$23,000+ HIGH

GPU-backed endpoints are rated HIGH confidence because the cost impact is unambiguous — zero invocations over 7 days on a GPU instance is definitively idle waste. CPU-backed endpoints at lower cost tiers are rated MEDIUM because they may be handling low but real traffic below the detection threshold.

Scanning for idle endpoints in CI

.github/workflows/ai-hygiene.yml AWS SageMaker — AI/ML waste scan 
- uses: aws-actions/configure-aws-credentials@v4
  with:
    role-to-assume: arn:aws:iam::${{ vars.AWS_ACCOUNT_ID }}:role/CleanCloudCIReadOnly
    aws-region: us-east-1

- uses: cleancloud-io/scan-action@v1
  with:
    provider: aws
    all-regions: 'true'
    category: ai            # Only AI/ML rules
    fail-on-confidence: HIGH # Fail on GPU endpoints with zero traffic
    output: json
    output-file: ai-findings.json

Required permissions: The rule needs sagemaker:ListEndpoints, sagemaker:DescribeEndpoint, and cloudwatch:GetMetricStatistics. These are all read-only. See the AWS setup guide for the full minimal IAM policy.

Azure: Idle Machine Learning Compute Clusters

Azure Machine Learning compute clusters are the training backbone for AML workflows. Unlike Azure VMs, AML compute clusters are designed to scale — but they only scale to zero if you explicitly configure min_nodes: 0. Clusters provisioned with min_nodes: 1 or higher keep at least one node running permanently, even when no training jobs are active.

The billing trap

A cluster with min_nodes: 1 billing on a Standard_NC6 (6-core GPU, 56GB RAM) costs approximately $600–$900/month when idle — billed continuously regardless of whether any training jobs have run. Multi-node clusters with GPU backing can easily exceed $10,000/month at idle minimum node counts.

How the rule works

The azure.ml.compute.idle rule checks three conditions for each compute cluster:

  1. No active or queued training jobs in the last N days (default: 7)
  2. min_nodes is greater than zero (meaning the cluster can't scale to zero)
  3. The cluster is in Succeeded or Running state (not already deallocated)

All three conditions must be met for a finding to be raised. A cluster with min_nodes: 0 that's been idle is not flagged — it's already scaled to zero and not billing.

GCP: Idle Vertex AI Prediction Endpoints

Vertex AI prediction endpoints deploy machine learning models for real-time inference. Like SageMaker, an idle Vertex AI endpoint continues to bill for the underlying compute resources — GPU-backed endpoints bill at full GPU instance rates even with zero prediction requests.

How the rule works

The gcp.vertex_ai.endpoint.idle rule uses Cloud Monitoring to check prediction/online_prediction_request_count over the idle window (default: 7 days). An endpoint with zero requests that has deployed model resources is flagged. Cost estimates are derived from the deployed model's machine type and accelerator configuration.

Machine type Approx. idle cost/month Confidence
n1-standard-4 (CPU) ~$130 MEDIUM
n1-standard-8 + T4 GPU ~$449 HIGH
n1-highmem-16 + V100 ~$2,200 HIGH
a2-highgpu-8g (8x A100) ~$23,000+ HIGH

Scanning in CI

- uses: google-github-actions/auth@v2
  with:
    workload_identity_provider: ${{ secrets.GCP_WORKLOAD_IDENTITY_PROVIDER }}
    service_account: ${{ secrets.GCP_SERVICE_ACCOUNT }}

Required GCP permissions: aiplatform.endpoints.list, aiplatform.endpoints.get, and monitoring.timeSeries.list. All read-only. The service account needs these on each scanned project.

Legitimate Exceptions: When Idle Isn’t Waste

Not every idle endpoint is waste. Some legitimate patterns look like idle endpoints from a traffic perspective:

  • Warm-standby endpoints — kept live for latency-sensitive workloads that must respond in milliseconds, where cold-start time is unacceptable
  • Batch-mode endpoints — called on a weekly or monthly schedule, genuinely idle between runs
  • Shadow deployment endpoints — receiving traffic in a production mirroring setup that may not log invocations the same way

These exceptions belong in your cleancloud.yaml — with a reason and an expiry date. That way they're documented, reviewable, and don't become permanent blind spots:

cleancloud.yaml — AI endpoint exception warm-standby, reviewed quarterly 
exceptions:
  - rule_id: aws.sagemaker.endpoint.idle
    resource_id: recommendation-engine-v3
    reason: "Warm standby — p99 latency SLA requires <100ms cold start"
    expires_at: "2026-07-01"

  - rule_id: azure.ml.compute.idle
    resource_id: monthly-retraining-cluster
    reason: "Monthly retraining job — idle between runs by design"
    expires_at: "2026-12-31"

Frequently Asked Questions

How much does an idle SageMaker endpoint cost?

An idle SageMaker endpoint bills the full on-demand rate for the underlying instance. A CPU-backed endpoint on ml.m5.xlarge costs roughly $140/month at idle. A GPU-backed endpoint on ml.g4dn.xlarge costs approximately $500/month. Larger GPU instances (ml.p4d.24xlarge) can reach $23,000+/month with zero traffic.

How do I detect idle SageMaker endpoints?

CleanCloud's aws.sagemaker.endpoint.idle rule checks InvocationCount via CloudWatch over a configurable idle window (default 7 days). Zero invocations flags the endpoint. GPU-backed instances are rated HIGH confidence; CPU-backed instances are rated MEDIUM.

Does Azure Machine Learning charge for idle compute clusters?

Yes. AML compute clusters with min_nodes > 0 continue to bill for the minimum node count even when no training jobs are running. The fix is to set min_nodes: 0 so the cluster scales to zero between jobs.

Can I use category=ai alongside hygiene rules?

Yes. Use category: all to run both hygiene and AI/ML rules in a single scan. This gives you the full picture — orphaned resources and idle AI infrastructure — in one pass.

Scan Your AI/ML Infrastructure Now

The fastest way to see what's idle in your environment is to run the AI/ML scan directly. It takes under 60 seconds and requires only read-only permissions:

AWS CloudShell or local terminal AI/ML rules only 
pip install cleancloud

# AWS — scan all regions for idle SageMaker endpoints
cleancloud scan --provider aws --all-regions --category ai

# Azure — scan all subscriptions for idle AML compute
cleancloud scan --provider azure --category ai

# GCP — scan all projects for idle Vertex AI endpoints
cleancloud scan --provider gcp --all-projects --category ai

For ongoing enforcement, add the weekly scheduled scan to your CI pipeline and set fail-on-confidence: HIGH. Every GPU endpoint with zero traffic will fail the build until it's remediated or explicitly suppressed with a reason and expiry date in cleancloud.yaml.
