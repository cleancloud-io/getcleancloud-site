Vertex AI Endpoint Running 3 Months After a PoC Ended:
$6,300 in Avoidable Spend

A GCP team ran a Vertex AI proof-of-concept with a V100-backed prediction endpoint. The proof-of-concept ended in January 2026. The endpoint was still running in late March. Seventy-eight days later, CleanCloud caught $6,300 in waste that nobody had noticed.

78 Days. Zero Requests. $6,300.

In January 2026, a platform team at a logistics company wrapped up a Vertex AI proof-of-concept. They’d been evaluating whether to migrate their route optimisation model from a self-managed TensorFlow serving setup to Vertex AI Online Prediction. The evaluation ran for six weeks. The conclusion was “not yet” — the latency profile didn’t meet their SLA requirements.

The decision was documented. The project was closed. The Vertex AI endpoint was not deleted.

The endpoint was named route-opt-poc-v2. It was running an n1-highmem-16 machine with a single NVIDIA V100 GPU. Every day it ran with zero prediction requests. Nobody checked. The GCP billing dashboard showed a steady ML infrastructure line item — but it looked exactly the same as it had during the PoC when traffic was real.

Why Vertex AI Billing Is Invisible in a Dashboard

Vertex AI Online Prediction endpoints bill for the deployed model's compute resources continuously — not per request. Once a model is deployed to an endpoint, the underlying machine type and accelerator configuration runs until the deployment is explicitly removed.

This is the fundamental billing contract: you pay for availability, not usage. When traffic is healthy, this is the right tradeoff — the endpoint must be ready to respond at any time. When traffic stops, it becomes silent waste.

The reason this pattern is hard to spot in billing dashboards is that the cost line looks identical whether the endpoint is serving real traffic or serving zero requests. There’s no “idle” signal in Cloud Billing. The only place that signal exists is Cloud Monitoring — specifically the prediction/online_prediction_request_count metric.

The detection gap: GCP Cloud Billing shows you spend. Cloud Monitoring shows you traffic. Neither product tells you when spend and traffic have diverged. CleanCloud joins these two signals to find endpoints where billing is happening and traffic is zero.

How the Detection Rule Works

The gcp.vertex_ai.endpoint.idle rule:

  1. Lists all Vertex AI endpoints in each scanned project using the AI Platform API
  2. For each endpoint with deployed models, checks prediction/online_prediction_request_count via Cloud Monitoring over the idle window (default: 7 days)
  3. Flags endpoints with zero requests and classifies by risk based on machine type and accelerator

Catching It in CI: The Weekly Scan

The setup that would have caught this weeks earlier — before $6,300 accumulated — is a weekly GitHub Actions scan against all GCP projects:

.github/workflows/gcp-ai-waste.yml GCP Vertex AI — weekly idle scan 
name: GCP AI/ML waste scan
on:
  schedule:
    - cron: '0 8 * * 1'   # Every Monday at 08:00 UTC

jobs:
  scan:
    runs-on: ubuntu-latest
    steps:
      - uses: google-github-actions/auth@v2
        with:
          workload_identity_provider: ${{ secrets.GCP_WORKLOAD_IDENTITY_PROVIDER }}
          service_account: ${{ secrets.GCP_SERVICE_ACCOUNT }}

      - uses: cleancloud-io/scan-action@v1
        with:
          provider: gcp
          all-projects: 'true'
          category: ai
          fail-on-confidence: HIGH
          fail-on-cost: '500'
          output: json
          output-file: gcp-ai-findings.json
          artifact-name: gcp-ai-report

With this setup, the route-opt-poc-v2 endpoint would have been flagged on its first Monday scan after going idle — seven days after the last request. The build would have failed with a HIGH confidence finding costing $2,420/month. Total waste that would have been avoided: $6,300 minus the cost of one week’s run — around $565.

Required GCP Permissions

The rule needs read-only access to Vertex AI and Cloud Monitoring. Assign these roles to the service account:

  • roles/aiplatform.viewer — or individual aiplatform.endpoints.list + aiplatform.endpoints.get
  • roles/monitoring.viewer — or individual monitoring.timeSeries.list

For scanning all projects via all-projects: true, the service account also needs roles/browser at the organisation level to enumerate projects.

What Happened: The Same Day Resolution

The logistics team was running CleanCloud for the first time when this finding appeared. The platform engineer who ran the scan confirmed the endpoint name with the data science lead who’d run the PoC. “Oh god, I forgot to delete that.”

The endpoint was undeployed and deleted within the hour. The team added a policy to their cleancloud.yaml requiring all PoC endpoints to be suppressed with an explicit expiry date — or flagged and deleted. Any endpoint that appears in the scan without a suppression entry fails the CI build.

cleancloud.yaml — PoC endpoint exception pattern 30-day expiry on all experimental endpoints 
# Policy: all PoC/experimental endpoints must have an exception with expiry <= 30 days
exceptions:
  # Active PoC — being actively evaluated, expires in 3 weeks
  - rule_id: gcp.vertex_ai.endpoint.idle
    resource_id: llm-routing-poc-v1
    reason: "Active PoC evaluation — traffic starts January 15"
    expires_at: "2026-02-07"    # 3 weeks — forces a conscious renewal

When the expiry passes, the endpoint either has real traffic (and the rule won't flag it anyway) or it's still idle (and the CI build fails, forcing deletion or renewal of the exception). The PoC endpoint pattern that cost $6,300 can’t silently accumulate for months again.

Frequently Asked Questions

Does Vertex AI charge when an endpoint has zero prediction requests?

Yes. Vertex AI prediction endpoints bill for the deployed model’s compute resources continuously — machine type, accelerators, and minimum replica count — regardless of request volume. Zero requests does not mean zero billing.

How do I detect idle Vertex AI endpoints automatically?

Run cleancloud scan --provider gcp --all-projects --category ai or use the cleancloud-io/scan-action GitHub Action on a weekly schedule with fail-on-confidence: HIGH. Endpoints with zero online_prediction_request_count over 7 days are flagged.

How do I delete a Vertex AI endpoint that’s no longer needed?

First undeploy all models from the endpoint, then delete the endpoint resource. Via gcloud: gcloud ai endpoints undeploy-model ENDPOINT_ID --deployed-model-id MODEL_ID --region REGION, then gcloud ai endpoints delete ENDPOINT_ID --region REGION.
