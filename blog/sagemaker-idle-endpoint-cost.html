How We Found $25,000 in Idle SageMaker Endpoints
Nobody Knew Were Running

A data science team deployed a fraud detection model for an internal hackathon. The hackathon ended. The model was deprecated. The SageMaker endpoint wasn't. Six weeks later, CleanCloud flagged $25,312 in idle GPU billing nobody had noticed.

The Hackathon That Kept Billing

It started with a routine weekly scan. A fintech platform team had just added CleanCloud to their CI pipeline — nothing fancy, a scheduled GitHub Action running every Monday morning against their AWS Organization. The scan took four minutes. The results were not routine.

Buried in the JSON output were eight SageMaker endpoints spread across three regions. All of them had zero invocations. Two of them were backed by ml.p3.8xlarge instances — four NVIDIA Tesla V100 GPUs each, running 24 hours a day, seven days a week, for the past six weeks.

The platform team lead looked up the endpoint names. fraud-detection-hackathon-v3. txn-scoring-prod-shadow. Names that sounded important — production-adjacent, clearly not temporary. She searched Slack. She found a message from six weeks earlier: “great hackathon everyone! the model is in v3 now, deprecated v1 and v2, will clean up infra this week.”

The infra cleanup never happened. The data scientist who wrote it had moved to a different team. The endpoints kept running. The bill kept accumulating.

“The message said ‘will clean up infra this week.’ That was six weeks ago. The endpoints were still running.”

Total waste over the six-week period: $25,312. Caught on day 43. Would have continued indefinitely if the scan hadn't flagged it.

Why SageMaker Hides This From You

This story is not unusual. The reason idle SageMaker endpoints are such a persistent source of waste comes down to how SageMaker's billing model works — and how it differs from what most engineers expect.

When you deploy a SageMaker endpoint, AWS provisions EC2 compute capacity behind a load balancer and a managed API. That compute runs continuously, regardless of whether it receives any requests. Unlike Lambda (where you pay per invocation), unlike Fargate (where containers start on demand), a SageMaker real-time endpoint is always-on infrastructure.

The billing consequences:

  • An ml.g4dn.xlarge endpoint (entry-level GPU) costs ~$504/month idle — every month, every day, whether it gets one request or zero
  • An ml.p3.2xlarge endpoint costs ~$2,800/month idle
  • An ml.p3.8xlarge endpoint (four V100 GPUs) costs ~$5,600/month idle
  • An ml.p4d.24xlarge endpoint (8x A100 GPUs) can exceed $23,000/month idle

None of this shows up as an anomaly in a cost dashboard. It looks exactly like a healthy, running endpoint. The only signal that it's idle is the invocation count — which lives in CloudWatch, not in Cost Explorer.

The detection gap: Cost dashboards show you spending. CloudWatch shows you traffic. Neither tool tells you when spending is happening with zero traffic. That's the gap CleanCloud's aws.sagemaker.endpoint.idle rule fills.

How the Detection Rule Works

The aws.sagemaker.endpoint.idle rule does three things:

  1. Lists all SageMaker endpoints in the account/region using sagemaker:ListEndpoints filtered to InService status
  2. Checks InvocationCount via CloudWatch Metrics (cloudwatch:GetMetricStatistics) over the idle window (default: 7 days)
  3. Classifies by risk based on instance type: GPU-backed endpoints are rated HIGH confidence (zero invocations on a GPU instance is definitively wasteful), CPU-backed endpoints at lower cost tiers are rated MEDIUM

The cost estimate comes from the instance type's on-demand pricing in the endpoint's region, multiplied by 720 hours/month. It's deliberately conservative — it doesn't assume multi-AZ deployment or data transfer costs, so the real figure is often higher.

Catching It in CI: The Setup That Caught $25K

The fintech team's setup was simple. A scheduled workflow running every Monday, scanning their AWS Organization across all regions, with category: ai to focus on AI/ML rules and fail-on-confidence: HIGH to fail the job when GPU endpoints are found idle:

.github/workflows/ai-waste.yml The exact setup that caught $25K 
name: AI/ML waste scan
on:
  schedule:
    - cron: '0 8 * * 1'   # Every Monday at 08:00 UTC

jobs:
  scan:
    runs-on: ubuntu-latest
    permissions:
      id-token: write
      contents: read
    steps:
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::${{ vars.AWS_ACCOUNT_ID }}:role/CleanCloudCIReadOnly
          aws-region: us-east-1

      - uses: cleancloud-io/scan-action@v1
        with:
          provider: aws
          org: 'true'          # Scan entire AWS Organization
          all-regions: 'true'
          category: ai          # Only AI/ML rules
          fail-on-confidence: HIGH
          fail-on-cost: '500'
          output: json
          output-file: ai-findings.json
          artifact-name: ai-waste-report

That's it. The scan ran for the first time on a Monday morning. It failed the job. The platform lead got the notification, opened the artifact, and found the eight endpoints. Four minutes of CI time caught $25,312 in waste.

What Happened Next

The platform team contacted the data science lead. The endpoints were confirmed abandoned — the hackathon models had been superseded by a new architecture and nobody had cleaned up the serving infrastructure. The endpoints were deleted the same day the scan ran.

One endpoint — txn-scoring-prod-shadow — turned out to be a legitimate shadow deployment that was still needed. The team added it to their cleancloud.yaml with a reason and an expiry date:

cleancloud.yaml — legitimate exception suppressed documented, expiring 
exceptions:
  - rule_id: aws.sagemaker.endpoint.idle
    resource_id: txn-scoring-prod-shadow
    reason: "Shadow deployment for v4 rollout — receives mirrored traffic, low volume"
    expires_at: "2026-06-30"

The other seven endpoints were deleted. Monthly saving: ~$21,700. The scan now runs every Monday. Any new idle GPU endpoint fails the build within seven days of going idle — before it can accumulate significant waste.

Required IAM Permissions

The rule requires read-only access to SageMaker and CloudWatch. Add these actions to your CleanCloud IAM policy:

IAM policy additions for SageMaker idle detection read-only, no writes 
"Action": [
  "sagemaker:ListEndpoints",
  "sagemaker:DescribeEndpoint",
  "cloudwatch:GetMetricStatistics"
],
"Effect": "Allow",
"Resource": "*"

Frequently Asked Questions

Does SageMaker charge when an endpoint has zero traffic?

Yes. SageMaker endpoints are always-on infrastructure. The EC2 instance behind the endpoint runs continuously regardless of traffic. AWS bills at full on-demand rates including GPU capacity — even with zero invocations.

How do I detect idle SageMaker endpoints automatically?

Run cleancloud scan --provider aws --all-regions --category ai or use the cleancloud-io/scan-action GitHub Action on a weekly schedule with fail-on-confidence: HIGH. GPU-backed endpoints with zero invocations over 7 days will fail the build.

What's the cheapest GPU endpoint I might have forgotten?

An ml.g4dn.xlarge — the smallest GPU instance available in SageMaker — costs approximately $504/month at idle. If you have one left over from a hackathon or proof-of-concept, it's costing you around $6,000/year with zero traffic.
