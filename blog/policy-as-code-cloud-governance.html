Policy-as-Code Cloud Governance:
Stop Managing Exceptions in Spreadsheets

Cloud hygiene exceptions managed in spreadsheets or Jira comments rot fast. Policy-as-code puts your rules, exceptions, and thresholds in git alongside your infrastructure — reviewable, auditable, and expiring automatically.

S
Suresh · CleanCloud

The Exceptions Spreadsheet Nobody Trusts

Every team that runs cloud hygiene scans eventually creates an exceptions list. A bastion host that's intentionally stopped. A dev database that's been idle for 45 days but is still in active use. A NAT gateway with no traffic because the workload is seasonal. Legitimate exceptions, all of them.

The problem isn't having exceptions. The problem is how most teams manage them. A shared spreadsheet. A Jira epic. A Slack pinned message. An undocumented flag in a scan config that nobody remembers adding. Three months later, nobody knows which exceptions are still valid, who approved them, or when they were supposed to expire.

The exceptions spreadsheet is the FinOps equivalent of suppressing a security alert by commenting it out in the config. The signal is still there. The waste is still running. You've just stopped seeing it.

“An exception with no expiry date isn't an exception. It's a permanent blind spot with extra steps.”

Policy-as-code solves this by making exceptions first-class, version-controlled, and self-expiring. Every exception has a reason, an expiry date, and a git commit behind it. The moment the expiry passes, the finding resurfaces — automatically, without anyone having to remember to check.

What Policy-as-Code Means for Cloud Hygiene

Policy-as-code is a well-established pattern in security (OPA, Sentinel, Checkov) and infrastructure (Terraform guardrails, AWS Service Control Policies). The core idea is simple: instead of configuring your compliance rules through a UI, a spreadsheet, or tribal knowledge, you express them in a versioned file that lives alongside your infrastructure code.

For cloud hygiene, policy-as-code means a single cleancloud.yaml file committed to your repository root that controls:

  • Defaults — minimum confidence and cost thresholds that apply to every scan
  • Exceptions — specific resources suppressed, with a reason and an expiry date
  • Thresholds — CI enforcement rules: fail on HIGH confidence, fail if waste exceeds $500/month
  • Tag filtering — ignore entire environments (e.g., env:dev) without suppressing individual resources
  • Rule config — tune idle thresholds, minimum cost per rule, or disable rules that don't apply to your architecture

CleanCloud picks this file up automatically if it's in the repo root. No flag needed. No pipeline configuration to update. Just commit the file.

Anatomy of a cleancloud.yaml

Here's a complete, annotated policy config covering the most common patterns:

cleancloud.yaml committed to repo root 
# Defaults apply to every scan run from this repo
defaults:
  confidence: MEDIUM          # Only show MEDIUM and above
  min_cost: 10               # Ignore findings under $10/month

# CI enforcement thresholds — override with --fail-on-confidence flag if needed
thresholds:
  fail_on_confidence: HIGH
  fail_on_cost: 500           # Fail CI if any single finding costs > $500/month

# Exceptions: specific resources with reason + expiry
exceptions:
  # Bastion host — started on demand, not idle
  - rule_id: aws.ec2.instance.stopped
    resource_id: i-0abc1234567890def
    reason: "Bastion host — started on demand via SSM"
    expires_at: "2026-12-31"

  # Seasonal NAT gateway — traffic spikes quarterly
  - rule_id: aws.ec2.nat_gateway.idle
    resource_id: nat-0def1234567890abc
    reason: "Quarterly batch pipeline — idle between runs"
    expires_at: "2026-07-01"
    account_id: 111111111111   # Scoped to one account in multi-account orgs

  # Glob pattern — suppress all dev volumes across all accounts
  - rule_id: aws.ec2.volume.unattached
    resource_id: vol-dev-*
    reason: "Dev environment volumes — ephemeral by design"
    expires_at: "2027-01-01"

# Tag-based filtering — ignore entire environments without suppressing individually
scan:
  ignore_tags:
    - env:dev          # Ignore anything tagged env=dev
    - temporary        # Ignore anything with a 'temporary' tag (any value)

# Per-rule tuning
rules:
  aws.ec2.instance.stopped:
    idle_days: 60       # Raise threshold from default 30 days to 60
  aws.cloudwatch.log_group.no_retention:
    enabled: false      # Not relevant for our architecture

Exceptions: What Works, What Doesn't

Always include a reason

A reason string is optional in the schema but should be mandatory in your team's practice. Six months from now, nobody will remember why vol-0abc1234 was suppressed. The reason field is your future self's documentation.

Always set an expiry date

Exceptions without expires_at never resurface. Use expires_at even if you're confident the exception is permanent — set it a year out and let the renewal decision be a conscious act. When the date passes, CleanCloud stops suppressing the finding. The resource appears in scan output again and can fail CI.

Tip: Use --explain in CI to print suppression reasons alongside filtered findings. This makes it easy to audit which resources are currently suppressed and why — right in the CI log.

Use glob patterns for environments

For exceptions that cover a class of resources rather than a single ID, use glob patterns. This is much cleaner than listing every dev volume individually:

cleancloud.yaml — glob exception matches vol-dev-* across all accounts 
exceptions:
  - rule_id: aws.ec2.volume.unattached
    resource_id: vol-dev-*
    reason: "Dev environment volumes — ephemeral by design"
    expires_at: "2027-01-01"

Scope to a specific account in multi-account orgs

When you're scanning an AWS Organization, an exception for a resource in account A shouldn't suppress a resource with the same ID in account B. Use account_id to scope it:

cleancloud.yaml — account-scoped exception only applies in account 111111111111 
exceptions:
  - rule_id: aws.ec2.nat_gateway.idle
    resource_id: nat-0def1234567890abc
    reason: "Quarterly batch pipeline — idle between runs"
    expires_at: "2026-07-01"
    account_id: 111111111111

Enforcement Thresholds: Fail CI Intelligently

The most powerful part of policy-as-code for cloud hygiene is defining exactly when your CI pipeline should fail. Two thresholds give you precise control:

Threshold What it does When to use
fail_on_confidence: HIGH Fails CI if any finding has HIGH confidence Start here — only fails on very clear waste
fail_on_confidence: MEDIUM Fails CI if any finding is MEDIUM or above Use after you've cleared HIGH findings and tuned exceptions
fail_on_cost: 500 Fails CI if any single finding costs >$500/month Catches expensive waste regardless of confidence level
fail_on_findings: true Fails CI on any finding at all Too aggressive for most teams — combine with tight exceptions instead

The recommended rollout sequence is: start with fail_on_confidence: HIGH, suppress legitimate exceptions, then tighten to MEDIUM once the signal is clean. Cost-based thresholds are complementary — they catch expensive anomalies even when confidence is ambiguous.

Tag Filtering: Ignore Environments Without Suppression Lists

Individual resource exceptions are the right tool for specific, known resources. But for ignoring entire environments — dev clusters, sandbox accounts, temporary experiment infrastructure — tag filtering is much cleaner. Instead of maintaining a list of hundreds of resource IDs, you tag resources with env:dev and let CleanCloud skip them entirely:

cleancloud.yaml — tag-based environment filtering no resource IDs needed 
scan:
  ignore_tags:
    - env:dev          # key:value — only ignores exact match
    - env:staging      # ignore staging too
    - temporary        # key only — ignores any value of 'temporary' tag

What tag filtering does not do: it doesn't suppress untagged resources. A resource with no tags at all will still appear in scan output. This is intentional — untagged resources are often the highest-priority finding because they have no owner by definition. Use the aws.resource.untagged rule to enforce tagging compliance separately.

The Full CI Pattern: Policy Config + Scan Action

Once your cleancloud.yaml is committed, integrating it into GitHub Actions requires no additional configuration. The scan action auto-detects the file from the repo root after checkout:

.github/workflows/cloud-hygiene.yml Policy config auto-detected from repo root 
name: Cloud hygiene
on:
  schedule:
    - cron: '0 8 * * 1-5'   # Weekdays at 08:00 UTC
  push:
    branches: [main]

jobs:
  scan:
    runs-on: ubuntu-latest
    permissions:
      id-token: write
      contents: read
    steps:
      - uses: actions/checkout@v4   # Required — makes cleancloud.yaml available

      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::${{ vars.AWS_ACCOUNT_ID }}:role/CleanCloudCIReadOnly
          aws-region: us-east-1

      - uses: cleancloud-io/scan-action@v1
        with:
          provider: aws
          all-regions: 'true'
          # cleancloud.yaml is picked up automatically from repo root
          # fail_on_confidence and fail_on_cost thresholds come from cleancloud.yaml
          output: json
          output-file: scan-results.json
          explain: 'true'   # Print suppression reasons in CI log
          artifact-name: cleancloud-report

When the scan runs, you'll see exactly which resources were suppressed and why, which findings remain, and whether any thresholds were breached. The --explain flag makes the suppression list auditable in every CI run.

Per-Environment Configs

For organisations running separate CI pipelines per environment — prod, staging, sandbox — you can maintain separate policy configs and reference them explicitly:

CI step — explicit config path override auto-detection 
- uses: cleancloud-io/scan-action@v1
  with:
    provider: aws
    all-regions: 'true'
    config: configs/cleancloud.prod.yaml   # Strict: fail on HIGH confidence

# In a separate staging job:
- uses: cleancloud-io/scan-action@v1
  with:
    provider: aws
    all-regions: 'true'
    config: configs/cleancloud.staging.yaml # Relaxed: report only

The typical pattern is a strict policy for production (fail on HIGH, fail above $500/month) and a relaxed policy for staging (report-only, or fail only on very expensive findings). Sandbox environments often get tag-filtered out entirely with ignore_tags: [env:sandbox].

Why This Is Governance, Not Just Config

The word “governance” gets overloaded in the cloud space, but for policy-as-code it has a specific meaning. When your cloud hygiene rules, exceptions, and thresholds live in git:

  • Every exception has a git author. You know who approved it and when.
  • Every exception has a history. git log cleancloud.yaml shows you every change to your suppression list.
  • Exceptions require a PR. Adding a new exception goes through code review, the same as any infrastructure change.
  • Exceptions expire automatically. No manual audit required. The scan surfaces the finding again on the expiry date.
  • Thresholds are change-controlled. Relaxing a threshold from HIGH to MEDIUM requires a PR, not a click in a dashboard.

This is the property that makes policy-as-code genuinely different from a Jira ticket or a spreadsheet row: you cannot silently change it. Every modification to your policy is a commit — and in regulated environments, that commit is your audit trail.

Frequently Asked Questions

What is policy-as-code for cloud governance?

Policy-as-code means defining your cloud hygiene rules, exceptions, and thresholds in a version-controlled file — like cleancloud.yaml — rather than in spreadsheets or tickets. Every exception has a reason, an expiry date, and a git commit. Changing the policy goes through code review.

How do cloud hygiene exceptions work in CI?

CleanCloud reads your cleancloud.yaml from the repo root (auto-detected after actions/checkout). Exceptions suppress specific findings for their configured duration. When an exception expires, the finding resurfaces and can fail the CI build again — forcing a conscious renewal decision.

What happens when a policy exception expires?

CleanCloud stops suppressing the finding. The resource appears in scan output again. If it meets the fail_on_confidence or fail_on_cost threshold, the CI build fails. The team must either renew the exception (with a new reason and expiry) or remediate the underlying resource.

Can I use policy-as-code with multi-account AWS Organizations?

Yes. Use the account_id field on any exception to scope it to a specific account. Global exceptions (without account_id) apply across all scanned accounts. This lets you suppress a resource in one account without accidentally suppressing a resource with the same ID in a different account.

Getting Started in Two Steps

The fastest path to policy-as-code governance is to run a scan first, identify which findings are legitimate exceptions, then write the config. You don't have to define your entire policy upfront.

Step 1 — discover what's there AWS CloudShell or local terminal 
pip install cleancloud
cleancloud scan --provider aws --all-regions --output json --output-file findings.json
Step 2 — commit your policy config repo root 
# cleancloud.yaml
defaults:
  confidence: MEDIUM
  min_cost: 10

thresholds:
  fail_on_confidence: HIGH

exceptions:
  - rule_id: aws.ec2.instance.stopped
    resource_id: i-0abc1234567890def
    reason: "Bastion host — on-demand via SSM"
    expires_at: "2026-12-31"

From the first scan, you have a baseline. From the committed config, you have governance. The CI pipeline runs daily and fails on anything new that clears your threshold. Exceptions require a PR. Thresholds are change-controlled. That's policy-as-code cloud governance — and it took two steps to set up.
